1. Volatile-保证线程可见性

volatile 关键字，使一个变量在多个线程间可见

执行下面代码, 分析执行结果: 

public class HelloVolatile {
    volatile boolean running = true; //对比一下有无volatile的情况下，整个程序运行结果的区别
    void m() {
        System.out.println("m start");
        while(running) {
            /*try {
                TimeUnit.MILLISECONDS.sleep(10);
            }catch (InterruptedException e){
                e.printStackTrace();
            }*/
        }
        System.out.println("m end!");
    }

    public static void main(String[] args) {
        HelloVolatile t = new HelloVolatile();

        /*new Thread(new Runnable() {
            @Override
            public void run() {
                t.m();
            }
        });*/
        new Thread(t::m, "t1").start();

        try {
            TimeUnit.SECONDS.sleep(1);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        t.running = false;
    }
}

不适应volatile修饰running:

在上面代码中，running是存在于堆内存的t对象中

当线程t1开始运行的时候，会把running值从内存中读到t1线程的工作区，在运行过程中直接使用这个copy，并不会每次都去读取堆内存，这样，当主线程修改running的值之后，t1线程感知不到，所以不会停止运行

使用volatile修饰running:

使用volatile，将会强制所有线程都去堆内存中读取running的值

如下图所示:

  • A B线程都用到堆内存中的一个变量flag

  • java线程执行过程中会将堆内存中的flag=true拷贝一份到自己线程的工作空间

  • 如果线程B修改flag=false后将值写回堆内存中

  • 但线程A未必会去堆内存中重新读取flag的值, 而是继续使用自己工作空间中的副本值, 则会导致程序执行出现错误.

  • 使用Volatile对flag进行修饰, 会让所有线程都会读到flag的修改值

底层实现是MESI, CPU的缓存一致性协议, 多核CPU,保证缓存数据一致性

但是, volatile并不能保证多个线程共同修改running变量时所带来的不一致问题，也就是说volatile不能替代synchronized

2.DCL单例— 禁止指令重排序

2.1 单例设计模式

单例模式（Singleton Pattern）是 Java 中最简单的设计模式之一。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。

这种模式涉及到一个单一的类，该类负责创建自己的对象，同时确保只有单个对象被创建。这个类提供了一种访问其唯一的对象的方式，可以直接访问，不需要实例化该类的对象。

2.2 单例模式饿汉式

类加载到内存后, 就诗意化一个单例, JVM保证线程安全简单实用, 推荐使用

唯一缺点: 不管用到与否, 类装载时就完成实例化

Class.forName(“”)但是如果我们不使用的化为什么要实例化

public class Singleton01 {
    private static final Singleton01 INSTANCE = new Singleton01();

    private Singleton01() {};

    public static Singleton01 getInstance(){
        return INSTANCE;
    }
    public void m(){
        System.out.println("m");
    }
    public static void main(String[] args) {
        Singleton01 s1 = Singleton01.getInstance();
        Singleton01 s2 = Singleton01.getInstance();
        System.out.println(s1 == s2);
    }
}

2.3 单例模式懒汉式

lazy loading 也称懒汉式

虽然达到了按需初始化的目的，但却带来线程不安全的问题

public class Singleton02 {
    private static Singleton02 INSTANCE;

    private Singleton02() {
    }
    public static Singleton02 getInstance() {
        if (INSTANCE == null) {
            try {
                Thread.sleep(1);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            INSTANCE = new Singleton02();
        }
        return INSTANCE;
    }

    public void m() {
        System.out.println("m");
    }
    public static void main(String[] args) {
        for(int i=0; i<100; i++) {
            new Thread(()->
                    System.out.println(Singleton02.getInstance().hashCode())
            ).start();
        }
    }
}

2.4 线程不安全的问题

多线程高并发环境下, 如果第一个线程判断INSTANCE==null但实例化尚未完成,第二线程进来也判断INSTANCE为空,会导致对象实例化出现问题.

可以通过synchronized解决，但也带来效率下降

public class Singleton03 {
    private static Singleton03 INSTANCE;

    private Singleton03() {
    }

    public static synchronized Singleton03 getInstance() {
        if (INSTANCE == null) {
            try {
                Thread.sleep(1);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            INSTANCE = new Singleton03();
        }
        return INSTANCE;
    }

    public void m() {
        System.out.println("m");
    }

    public static void main(String[] args) {
        for(int i=0; i<100; i++) {
            new Thread(()->{
                System.out.println(Singleton03.getInstance().hashCode());
            }).start();
        }
    }
}

2.5 懒汉式synchronized优化

减小锁粒度, 代码如下:

public class Singleton04 {
    private static Singleton04 INSTANCE;

    private Singleton04() {}

    public static Singleton04 getInstance() {
        if (INSTANCE == null) {
            //妄图通过减小同步代码块的方式提高效率，然后不可行
            synchronized (Singleton04.class) {
                try {
                    Thread.sleep(1);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                INSTANCE = new Singleton04();
            }
        }
        return INSTANCE;
    }

    public void m() {System.out.println("m");}

    public static void main(String[] args) {
        for(int i=0; i<100; i++) {
            new Thread(()->{
                System.out.println(Singleton04.getInstance().hashCode());
            }).start();
        }
    }
}

但是这种方法还是可能会导致进行2次初始化.

  1. 线程1判断INSTANCE == null, 拿到锁后执行下面代码, 初始化还未完成
  2. 线程2判断INSTANCE == null, 等待线程1释放锁
  3. 线程1初始化完成后释放锁资源,线程2拿到锁资源后执行对象的初始化
  4. 会导致对象被初始化2次,违背单例设计模式原则

2.5 Double Check Lock

为解决2.4出现的问题, 使用双重校验解决

public class Singleton05 {
    private static Singleton05 INSTANCE; //JIT

    private Singleton05() {
    }

    public static Singleton05 getInstance() {
        //省略业务逻辑
        if (INSTANCE == null) {
            //双重检查
            synchronized (Singleton05.class) {
                if(INSTANCE == null) {
                    try {
                        Thread.sleep(1);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                    INSTANCE = new Singleton05();
                }
            }
        }
        return INSTANCE;
    }

    public void m() {
        System.out.println("m");
    }

    public static void main(String[] args) {
        for(int i=0; i<100; i++) {
            new Thread(()->{
                System.out.println(Singleton05.getInstance().hashCode());
            }).start();
        }
    }
}

使用这种方法已经可以解决很多高并发场景下的单例加载问题,但是在超高超高并发情况下还是可能会出现一些问题.

2.6 volatile, 禁止指令重排序

INSTANCE = new Singleton();

在线程对对象进行实例化的过程种,JVM实例化对象分为3个步骤:

  1. 为对象申请内存空间
  2. 为成员变量进行初始化
  3. 将内存中的内容赋值给INSTANCE

在对成员变量初始化的过程中会先进行实例化

第一步: 实例化过程中成员变量的值是默认值, 例如 int类型变量a = 0;

第二步: 对a初始化, 假如我们为a设置的值为8, 那么这一步会对a进行赋值 : a = 8 ;

第三步: 将初始化完成的 a=8 赋值给INSTANCE

如果发生指令重排序, a还没有初始化完成,也就是还没有进行第二部就执行了第三步,那么会将a实例化时的默认值也就是a=0赋值给INSTANCE; 假如有线程2使用该对象, 并且INSTANCE已经有值存在, 线程2不会再进行对象初始化, 而是使用INSTANCE的值, 可能会导致程序出错

使用volatile修饰Singleton06, 禁止指令重排序

public class Singleton05 {
    private static volatile Singleton06 INSTANCE; //JIT

    private Singleton05() {
    }

    public static Singleton06 getInstance() {
        //省略业务逻辑
        if (INSTANCE == null) {
            //双重检查
            synchronized (Singleton06.class) {
                if(INSTANCE == null) {
                    try {
                        Thread.sleep(1);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                    INSTANCE = new Singleton06();
                }
            }
        }
        return INSTANCE;
    }

    public void m() {
        System.out.println("m");
    }

    public static void main(String[] args) {
        for(int i=0; i<100; i++) {
            new Thread(()->{
                System.out.println(Singleton06.getInstance().hashCode());
            }).start();
        }
    }
}
